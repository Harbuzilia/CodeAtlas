# Repository Map (Tree-sitter) Skill

<context>
Этот скилл предоставляет агентам (особенно `contextscout` и `openagent`) метод для быстрого охвата всей архитектуры проекта через Abstract Syntax Tree (AST). 
Вместо того чтобы слепо искать через `grep` или читать сотни plików, агент может сгенерировать `repomap`, который использует алгоритм PageRank для оценки наиболее важных файлов, классов и сигнатур функций во всём проекте.
</context>

## Как сгенерировать Repository Map

Для извлечения AST-карты мы используем нативный CLI инструмента Aider. Это самая точная реализация алгоритма.

**Команда для генерации:**
Используй инструмент `bash` (если у тебя есть права) или попроси пользователя выполнить следующую команду в корне проекта:

```bash
export PROMPT_TOOLKIT_NO_CPR=1 PYTHONIOENCODING=utf-8 && uvx --from aider-chat aider --yes --no-auto-commits --model null --no-show-model-warnings --show-repo-map > .opencode/repomap.txt
```

**Обработка ошибок и адаптация к другим проектам:**
- **Требование Git и добавленных файлов:** Aider генерирует карту строго для файлов, добавленных в Git-индекс. Если команда пишет `Git repo: .git with 0 files`, значит файлы не проиндексированы. В таком случае выполни `git init` (если репозитория нет) и обязательно `git add .` (для добавления файлов в индекс, коммит при этом делать необязательно). При ошибке `prompt toolkit` убедись, что переменная `PROMPT_TOOLKIT_NO_CPR=1` передана в команду.
- **Исключение мусора (.gitignore / .aiderignore):** Aider автоматически уважает правила `.gitignore` (все ваши `node_modules`, `dist` и т.д. игнорируются изначально). Если в проекте есть папки, которые нужно скрыть от AST-карты, но они не в `.gitignore` (например, папки с логами или чужими референсами), создай файл `.aiderignore` в корне проекта и впиши туда эти папки (например `references/` или `.venv/`).
- Если команда долго "висит" - останови её.
- Если система не знает команду `uvx`, значит пакетный менеджер `uv` не настроен должным образом.
- Учти, что первая генерация может занять 10-30 секунд (в зависимости от размера репозитория), так как Tree-sitter должен проанализировать все исходники.

## Как использовать файл `.opencode/repomap.txt`

Как только файл сгенерирован (или если он уже существует), агенты должны **прочитать** его начало (`read(".opencode/repomap.txt")`), чтобы мгновенно понять высокоуровневую струкутуру.

### Формат вывода `repomap`
Сгенерированная карта выглядит примерно так:
```tree
src/
│
├─ auth/
│  └─ auth.ts
│     export class AuthService { ... }
│     export function login(user: User): Promise<Token>
│
├─ components/
│  └─ App.tsx
│     export const App = () => { ... }
```

### Правила использования карты (Для агентов)
1. **Не генерируй карту заново без необходимости:** Сначала проверь существование `.opencode/repomap.txt`. Если файл старше суток или проект сильно изменился — перегенерируй.
2. **Точечный поиск:** Используй `repomap`, чтобы найти, в каком именно файле определен нужный класс (например, `UserService`).
3. **Анализ зависимостей:** Если ты видишь функцию в `repomap`, но не знаешь её реализацию, переходи к точечному `read("path/to/file.ts")` для чтения конкретных строк.
4. **Не читай весь файл, если он огромный:** Если `repomap.txt` больше 1000 строк, используй `grep` по этому файлу для поиска нужных сущностей (например, `grep("class User", ".opencode/repomap.txt")`).
